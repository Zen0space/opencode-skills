## Known Patterns

### Documentation Types
| Type | Purpose | When to Use |
|------|---------|-------------|
| README | Project overview | Every project |
| API Docs | Endpoint details | Libraries, APIs |
| Tutorial | Step-by-step | New features |
| Guide | How-to | Common tasks |
| ADR | Architecture decisions | Major decisions |

### Writing Standards
| Standard | Description | Example |
|----------|-------------|---------|
| Active voice | Use direct language | "Click the button" not "The button should be clicked" |
| Present tense | Describe current state | "Returns value" not "Will return value" |
| Second person | Address reader directly | "You can..." not "Users can..." |
| Concise | Avoid unnecessary words | "Install" not "Perform an installation of" |

## Documentation Checklist by Level

### Level 1 - Novice
- [ ] README has basic sections
- [ ] Installation instructions clear
- [ ] Basic usage examples
- [ ] Simple code comments

### Level 2 - Apprentice
- [ ] API endpoints documented
- [ ] Configuration explained
- [ ] Error messages documented
- [ ] JSDoc comments added

### Level 3 - Practitioner
- [ ] Architecture overview written
- [ ] Design decisions recorded
- [ ] Tutorials created
- [ ] Complex examples provided

### Level 4 - Expert
- [ ] Contribution guide written
- [ ] Release process documented
- [ ] Migration guides created
- [ ] Advanced tutorials added

### Level 5 - Master
- [ ] Style guide comprehensive
- [ ] Multi-language support
- [ ] Documentation architecture planned
- [ ] Search/navigation optimized

### Level 6 - Grandmaster
- [ ] Documentation strategy defined
- [ ] Knowledge base structured
- [ ] Developer experience optimized
- [ ] Metrics tracked

## Code Example Standards

### Always Include
- Language identifier in code fence
- Complete runnable example
- Expected output shown
- Error handling demonstrated

### Never Include
- Outdated code
- Untested examples
- Placeholder code
- Non-working snippets

## Notes
- Level 1: Focus on basic documentation
- Higher levels unlock complex documentation
- Always test code examples before including
- Keep docs synchronized with code changes
